Appendix B
The Monorepo

The template's structure

A working prototype lives at sa-ai-loveable-monorepo. The shape below is what's in it today — not aspirational, not a wireframe. The rest of this section walks through why each piece exists and what it does.

The shape

sa-<project>/
├── CLAUDE.md                            # Root context for Claude Code
├── RULES.md                             # Shared rules — every agent reads
├── README.md
├── package.json                         # Bun workspaces
├── bunfig.toml                          # bun config (hoisted linker)
│
├── .claude/                             # Claude Code surface
│   ├── agents/                          #   specialized subagents
│   └── skills/                          #   documented patterns
│
├── .lovable/                            # Lovable surface
│   ├── RULES.md                         #   Lovable-only rules
│   ├── system.md                        #   design-system system prompt
│   └── rules/                           #   per-component & per-pattern rules
│
├── apps/
│   ├── web/                             # React + Vite (Lovable target)
│   │   ├── src/lib/api/                 #   adapter pattern lives here
│   │   └── eslint.config.js             #   bans @supabase/* imports
│   ├── api/                             # Fastify — CRUD & general backend
│   │   └── src/modules/<feature>/
│   └── python-api/                      # FastAPI — LLM & AI-specific
│
├── packages/
│   ├── api-sdk/                         # Runtime-agnostic business logic
│   │   └── src/<feature>/
│   └── ui/                              # @sa/ui — design system
│       └── lovable/                     #   source for .lovable/system.md + rules/
│
├── supabase/
│   └── functions/                       # Edge Functions (Lovable backend)
│       ├── _shared/sa-api-sdk/          #   vendored mirror of @sa/api-sdk
│       └── <feature>-<action>/
│
└── infra/                               # Project IaC — CDK
    ├── CLAUDE.md
    └── stacks/

A few things worth pointing out:

  • Two rule surfaces, one source of truth. RULES.md is shared. .lovable/RULES.md is Lovable-only. .lovable/system.md and .lovable/rules/ are generated from packages/ui/lovable/ via bun run sync:ui-rules, with check:ui-rules-sync enforcing zero drift in CI. Same pattern for supabase/functions/_shared/sa-api-sdk/, which mirrors packages/api-sdk/src/.
  • Same endpoint name, two runtimes. A feature like demo-personal exists in Fastify (apps/api/src/modules/demo/) and in Supabase Functions (supabase/functions/demo-personal/). Both call into @sa/api-sdk. The web client targets either backend by swapping only the base URL.
  • Bun is the only package manager. Matches Lovable's harness and gives us workspaces out of the box. The bunfig.toml at the root forces the hoisted linker so Vite + Rollup can resolve transitive dependencies inside the workspace.

Initial stack picks

The picks below are starting points, not commitments. Where there's a credible alternative, the engagement is the place to evaluate it on real work.

apps/web — Vite + React
Lovable's native target. Fast HMR, lean. Alternative: Vercel / Next.js if Saroj prefers the deploy story.
apps/api — Fastify
Lean, fast, ergonomic. Alternative: NestJS if SA wants the heavier opinionated framework with built-in DI.
apps/python-api — FastAPI
Default for LLM calls, embeddings, agent orchestration. Stable choice.
Package manager — Bun
Matches Lovable's harness. Workspace support out of the box. npm / pnpm / yarn are actively banned in RULES.md.

Stack selections should be re-evaluated as Lovable updates their default stack or as engineering requirements evolve. apps/web in particular tracks whatever Lovable generates on SA's plan today; if that default shifts (e.g., from React + Vite SPA to an SSR framework like TanStack Start), the template's web slot is the first thing to revisit.

The Rules

What Lovable is told to do

The biggest lever on the Product → Engineering handoff isn't where Lovable's code lives — it's what Lovable is told to write. The layered ruleset below is what makes Tier 1 possible.

The layered ruleset

Four mechanisms working together, each enforcing different things:

  • RULES.md at the repo root — shared rules every agent reads: package manager, portability rules, business-logic location, endpoint naming, design-system policy.
  • .lovable/RULES.md — Lovable-only rules: which directories are in scope vs. out of scope, the banned-import list, the env-var contract, agent behavior (don't run package commands, don't run stateful git).
  • .lovable/system.md + .lovable/rules/ — full design-system guidance (Atlas tokens, per-component, per-pattern), generated from packages/ui/lovable/ and CI-checked for drift.
  • ESLint enforcement for rules that can be enforced mechanically — no-restricted-imports blocks @supabase/* anywhere in apps/web/src/, so even if Lovable tries to add the dependency back, the lint step catches it.

The rules that matter most

Every rule below shrinks engineering's eventual rebuild surface area. Each one in the prototype today:

Calling backend services

UI/component code accesses backends only through the adapter at apps/web/src/lib/api/. Components never call fetch directly. No file in the web app imports @supabase/* — Edge Functions are reachable over plain HTTPS through the adapter. This means engineering never has to rip out Supabase SDK calls; the swap to a Fastify backend is a base-URL change, nothing else.

Business logic location

Cross-runtime business logic lives in packages/api-sdk/. It must be runtime-agnostic: no Deno, AWS SDK, Supabase SDK, or process.env imports. Anything that does I/O takes its adapter as an injected dependency. Per-runtime wrappers stay thin — parse runtime-specific inputs (HTTP, JWT, headers), call into the SDK, format the response.

Endpoint naming

Backend endpoints use flat kebab-case names: <feature>-<action>. The Fastify route path and the Supabase function directory name MUST match exactly. Same suffix, different base URL — that's what lets one client target either backend.

Design system

@sa/ui is the canonical source of components and design tokens. Use the primitives that exist; if a primitive is missing, add it to @sa/ui rather than to the consuming app. Don't recreate apps/web/src/components/ui/ — that's the shadcn-dump pattern the design system retired.

Evolving API contracts

For a project running in both runtimes during the Product phase: additive changes are cheap (a new function, a new optional parameter, a new SDK module merges cleanly). Breaking changes are fine when the feature needs them — Engineering absorbs the change at merge time. Gratuitous churn (renaming for taste, swapping shapes with no feature reason) is what to avoid — pure work for engineering, no product payoff.

The Adapter Pattern

How one piece of logic runs in both runtimes

The technical mechanism that makes Tier 2+ viable: dependency-injected adapters around a runtime-agnostic SDK. Same business logic, two thin wrappers — no divergence trap.

A worked example from the prototype: a "personal greeting" endpoint that looks up a user by ID, then returns a hello-by-name message. The shape generalizes to any endpoint that needs to do I/O.

1 · The SDK function (runtime-agnostic)
// packages/api-sdk/src/demo/composePersonalGreeting.ts

export interface ComposePersonalGreetingDeps {
  lookupUser: UserLookup;
}

export async function composePersonalGreeting(
  input: { userId: string },
  deps: ComposePersonalGreetingDeps,
): Promise<Result<{ message: string }>> {
  if (typeof input.userId !== "string" || input.userId.length === 0) {
    return { ok: false, status: 400, error: "userId required" };
  }

  const user = await deps.lookupUser(input.userId);
  if (!user) return { ok: false, status: 404, error: "user not found" };

  return { ok: true, value: { message: `Hello, ${user.displayName}!` } };
}

The orchestration — validate, look up, format — lives here. lookupUser is the runtime-specific piece, injected by the caller.

2 · The Fastify wrapper (engineering's runtime)
// apps/api/src/modules/demo/demo.service.ts

import { composePersonalGreeting, type UserLookup } from "@sa/api-sdk";

// In production this is a postgres query against SA's hosted DB.
const lookupUser: UserLookup = async (id) => pgClient.findUser(id);

export function buildPersonalGreeting(userId: string) {
  return composePersonalGreeting({ userId }, { lookupUser });
}
3 · The Supabase Edge Function wrapper (Lovable's runtime)
// supabase/functions/demo-personal/index.ts

import { composePersonalGreeting, type UserLookup } from "@sa/api-sdk";
import { corsPreflight, json } from "../_shared/http.ts";

// In production this is a supabase.from("users").select(...) call.
const lookupUser: UserLookup = (id) => supabaseClient.findUser(id);

Deno.serve(async (req) => {
  if (req.method === "OPTIONS") return corsPreflight();
  const userId = new URL(req.url).searchParams.get("userId") ?? "";
  const result = await composePersonalGreeting({ userId }, { lookupUser });
  if (!result.ok) return json(result.status, { error: result.error });
  return json(200, result.value);
});

What this buys. When engineering changes how the lookup behaves — caching, a different table, a join — they edit only the adapter. When the business logic changes — a new validation rule, a different formatting decision — they edit only the SDK. The two runtimes can't drift in their behavior because there is only one implementation of the behavior.

This is the answer to the "divergence trap" that historically makes dual-runtime work expensive. It doesn't make Tier 3+ free, but it makes the marginal cost of keeping both runtimes working scale with adapter complexity rather than with logic complexity.

What we've proven, per tier

The tiers from §2.5 with the testing detail filled in. Each row is: what's already validated in the prototype, what remains an open question, and what we'd test during the engagement.

00

Template / starting structure

Validated: the monorepo layout works end-to-end. Lovable's in-editor preview builds against it. Bun workspaces resolve correctly. The harness reports some cosmetic false positives (documented in .lovable/plan.md) but the actual build is healthy.

Open: none — already working.

Proven
01

Rules Lovable follows

Validated: in focused testing, Lovable followed the in-scope/out-of-scope rules, the banned-import list, the adapter pattern, the design-system rules, and the kebab-case endpoint naming. UI work in packages/ui followed the rules well once the rules + structure were in place.

Open: sustained adherence under real product-team workload. A multi-week run with several different feature shapes will tell us much more than a focused test.

Proven in initial testing
02

Clean handoff

Validated: architecturally proven. The SDK + adapter pattern means engineering's takeover is "swap stub adapters for real ones" — surgical, not a rebuild. The endpoint-naming and base-URL discipline mean the web client doesn't need to change.

Open: what to automate. A Claude Code skill or subagent that runs the conversion (provisioning real DB credentials, wiring SA auth, replacing stub adapters) would compress this further. We'd scope and build that during the engagement.

Architecturally proven
03

Dual-runtime through the Product phase

Validated: the parallel-branch workflow is documented (additive changes merge cheaply; breaking changes are fine but absorbed at merge time; gratuitous churn is what to avoid). The SDK pattern keeps logic from diverging.

Open: real-world cost. How often does Product change shape during active iteration? How much engineering time goes to merging breaking changes vs. additive ones? Measured on a real project, not estimated.

Architecturally proven
04

Dual-runtime for the project's life

Validated: nothing here is technically blocked. Both runtimes can coexist indefinitely.

Open: the maintenance economics. Every API change engineering makes has to be mirrored on the Supabase side. Probably not worth it for most products, but might be worth it for specific high-touch products where Product wants to keep prototyping against a real production system. Per-product judgment, not a blanket framework decision.

Open Questions

What we still need to test

The honest list of things this prototype hasn't yet stressed. Each is something the engagement would put through real use, not the kind of question we'd resolve in advance from first principles.

  • Rule adherence at scale. Focused testing showed Lovable follows the rules well. Sustained adherence — across many feature shapes, over weeks, under real product-team iteration — is the real test.
  • Complex Lovable workflows. The testing so far has been "is the shape preserved?" We haven't yet pushed Lovable through complex multi-step features with the rules in place. Some rules may turn out to be brittle under that load.
  • Cost of keeping the Lovable preview working long-term. Today the in-editor preview works on the prototype branch. Future Lovable harness changes could break it; how much engineering effort goes into keeping it green is unknown.
  • The shape and depth of handoff automation. Whether the Tier 2 conversion is best as a Claude Code skill, a CLI script, a subagent, or a documented manual flow — and how much it can do automatically vs. how much still requires engineering judgment.
  • How @sa/auth and the data-warehouse SDKs are consumed in Lovable. The prototype proves @sa/ui works as a workspace package; we expect the same to be true for @sa/auth and the service SDKs, but they haven't been built yet.
  • Lovable's ability to start from a vendored repo. Lovable's official position is that projects can only be started by prompt or by remixing a Lovable-side template. We have a working repo today; how to make that the default starting point for new SA projects (vs. having to re-create it inside Lovable every time) is an open mechanism question.
