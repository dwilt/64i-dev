Appendix C Supporting analysis

Infrastructure options
& the hybrid CDK case

Three structural options for where SA's infrastructure code lives, weighed against the framework's two compression levers. Final direction needs Cloud Engineering's input — this is the framing for that conversation.

Appendix C
Why This Matters

Why infrastructure is load-bearing

The AI Engineering → Cloud Engineering handoff is one of the two compression points the framework exists to fix. Where infrastructure code lives — and how it's structured — is the most direct lever on that handoff.

Today: AI Engineering ships a sandbox-shaped app to Cloud Engineering. Cloud Engineering writes Terraform in the central infrastructure repo. The handoff is high-effort because the IaC is being written from scratch after the app is done.

Target: the IaC for an app exists from the moment the framework provisions the repo. The handoff becomes a PR that touches finalized infra alongside finalized app code.

The structural choice — central monorepo, with-app, or hybrid — determines what that target actually looks like in practice. Each option below is real; each has costs that are real; the choice depends on Cloud Engineering's appetite for change.

Option A

Status quo: centralized Terraform

What SA does today. The central infrastructure repo owns all IaC across multiple teams, organized as teams/<team>/environments/<env>/ with a shared modules/ library. Cloud Engineering owns it.

What's good about it

  • Single audit surface. Every piece of cloud infrastructure SA owns has one place to be reviewed, security-scanned, and inventoried. Real value when compliance asks "where do we use KMS?" or "show me every public-facing ALB."
  • Shared modules library is genuinely reusable. The modules/ directory has 40+ entries covering common AWS primitives and per-app modules. New apps that fit existing patterns can wire up infra in a few hundred lines of wrapper.
  • Conventions enforced centrally. Tagging, IAM patterns, network topology, security tooling — all consistently applied because they all flow through the same module library.
  • Mature tooling. Terraform with S3-backed state, GitHub Actions for CI, golden-image base images, security-scanner integration. A real platform, not ad-hoc.

What's bad about it for the framework

  • The repo is a heavyweight cross-team coordination point. AI Engineering has to coordinate with Cloud Engineering to land a change. Today's friction is exactly that.
  • Infra and app code are physically separated. To understand "what does this app look like in prod?" a developer reads the app repo for code and the central infra repo for infra. Two repos, two PRs, two review trails. Easy to drift.
  • Cloud Engineering's scope is broader than this project. The central repo also contains data-warehouse infra, RPA infra, and so on. Blast radius of a change is bigger than it needs to be.
  • Module + wrapper is clever but heavy. A developer reading an environment's app config has to mentally hold the module's variables, the wrapper's locals, and the module's resources simultaneously. Effective once internalized; steep on-ramp.

What this would look like under the framework

Cloud Engineering keeps infrastructure as the source of truth. The framework's contribution is to pre-write the wrapper a new app needs, so the AI Engineering → Cloud Engineering handoff becomes "review and merge this PR" instead of "write that file from scratch."

This is the least disruptive path forward. Doesn't fix the infra-and-app-in-different-repos issue; does meaningfully shrink the handoff effort.

How it could be improved — even without changing tools

A lot of the per-app modules in the existing repo are doing the same job with the variable names changed. Compare modules/saai-field-lookup and modules/saai-idea-generator: both ~485 lines of Terraform, both standing up the same Supabase-on-EC2 stack (security groups, EC2 + IAM, ALB + listeners + target groups, Secrets Manager wiring, Route 53), with mostly resource-name changes between them. Five other saai-* modules in the repo follow the same shape at the same length.

Concrete numbers. Diffing the two modules side-by-side, roughly 85% of each file is structurally identical to the other. The differences are largely: which resource label gets which app's name, a handful of per-app port and FQDN values, and minor tagging variations. The actual architectural shape is one shape, copied five-plus times.

Most of that duplication can be removed by extracting higher-level reusable modules — what AWS docs call "L3 constructs" in CDK terminology, or just "opinionated wrapper modules" in Terraform terms. The shape would be:

  • An sa_app_stack module that takes a small set of parameters — app name, domain, instance type, whether the app needs Studio access, etc. — and renders the full security-group / EC2 / IAM / ALB / Secrets-Manager / Route-53 graph internally. Per-app modules become 30–50-line callers instead of 500-line copies.
  • Optional features as flags. "This app needs an S3 bucket" becomes files_bucket = true, not 80 lines of bucket + policy + encryption resources copy-pasted into yet another module. Same for "this app needs RDS access," "this app needs a Lambda for scheduled work," and so on.
  • The "shape" of an SA app becomes a single source of truth. When Cloud Engineering wants to add a new tagging rule or a new security baseline, they edit the L3 module once and every consumer picks it up at next apply. Today the same change requires touching each saai-* module independently — and forgetting one is silent drift.

This works in Terraform. A nested-module pattern (modules/sa-app-stack/ calling into smaller modules/sa-app-stack-alb/, modules/sa-app-stack-ec2/, etc., with the right outputs threaded through) achieves most of the same consolidation. It's not as elegant as CDK constructs but it's real, and it can land without changing the tool stack at all.

It works better in CDK. A CDK SaAppStack construct can use ordinary TypeScript — conditional logic, computed values, helper functions — to decide what to include based on a configuration object. The "if this app needs a files bucket, build one and wire its ARN into the task's IAM policy" pattern is one if in TypeScript, versus a count = var.files_bucket_enabled ? 1 : 0 dance threaded through every dependent resource in Terraform. This is the strongest argument for Option C below — but the consolidation win above is available either way.

Either way, the win is real. Cutting five 500-line modules down to a single 500-line L3 plus five 50-line wrappers isn't a 5x reduction in lines of code — it's a 5x reduction in places conventions can drift. Every audit, every patch, every new tagging rule, every observability default gets simpler. This is true regardless of which option SA picks; it's a refactor worth doing before the framework's per-app handoff pattern starts producing the sixth, seventh, and eighth copy.

Option B

Per-app infra, no central repo

Each app's repo owns its own IaC. No central infrastructure repo (or one used only for shared foundational infra like VPCs).

What's good about it

  • Locality. Read one repo, see the whole picture: app code, tests, Dockerfile, IaC, CI. PRs that change app behavior also change the infra in the same review.
  • No cross-team coordination tax for routine work. AI Engineering can iterate on infra without involving Cloud Engineering for every change.
  • Smaller blast radius. A bad merge affects one app, not "everything in infrastructure."
  • Matches modern developer mental model. Most engineering-grown developers expect infra-with-app; this is the default in modern AWS shops with CDK / Pulumi / SAM.

What's bad about it

  • Convention enforcement gets harder. Every app's IaC could wander off. Networking, IAM, tagging, observability — without something pulling these together, you accumulate N slightly-different versions and Cloud Engineering loses the central audit point.
  • Cross-app shared resources are awkward. VPCs, transit gateways, shared KMS keys — these live somewhere; if every app owns its own infra, the shared substrate has to live in some additional repo, partway back to where we started.
  • Cloud Engineering loses leverage. Today they touch every app's infra by definition. Under per-app, they only see what teams choose to involve them in.
  • Compliance and audit story degrades. "Show me every public-facing endpoint" goes from grep in one repo to crawling N repos.

For SA specifically, this is probably not the right destination at any phase — Cloud Engineering's role depends on a level of central oversight that pure per-app gives up. Useful as a thought experiment; not viable as a target. Worth considering as a component of the hybrid below.

Option C — Recommended Direction

Hybrid: per-app infra + shared CDK constructs

The shape the framework is drawn to. App repos own their own IaC code, but the IaC depends on a shared infrastructure package — likely CDK — that Cloud Engineering owns and ships.

How it would work

  • Cloud Engineering publishes a versioned package — @sa/infra or similar — exposing opinionated constructs for the SA-blessed primitives: a standard ECS Fargate service (tagging, log groups, IAM, security groups, ALB, monitoring all wired in); a standard Aurora cluster; a standard API Gateway endpoint; a scheduled task (ECS scheduled task or Lambda + EventBridge) with cross-cutting defaults.
  • App repos consume this package in their own infra/ directory. The app's IaC says "I want one of these, with these app-specific parameters" — much shorter and more readable than today's wrapper-calls-module.
  • Cloud Engineering owns the constructs (and their conventions); app teams own the composition (which constructs, with what parameters).
  • Cross-app shared substrate — VPCs, KMS, hosted zones, ECR, account-level IAM baselines — stays in a small central repo Cloud Engineering owns directly.

Why CDK rather than Terraform for this

Plausibly the technical choice that makes the hybrid actually work cleanly:

  • Native package semantics. CDK is TypeScript / Python; "shared package" is npm install or pip install. No Terraform module-source path gymnastics, no relative paths into a sibling repo, no submodule hell. Versioning is semver; breaking changes are signaled by major-version bumps.
  • Constructs are higher-leverage than modules. A CDK construct can wrap multiple AWS resources with conditional logic, cross-reference, and computation in a way Terraform modules can but more awkwardly. The existing per-app Terraform modules are doing this today; CDK would do it more naturally.
  • App developers already know TypeScript. AI Engineering is mostly TS/Python. Reading and modifying TS infrastructure code is closer to their daily skill than HCL.
  • It compounds with the rest of the framework. The template's infra/ directory becomes consistent with the rest of the codebase — same language, same toolchain, same ecosystem.

The honest counter-argument. Cloud Engineering's existing investment is in Terraform — modules, state management, golden Docker images, security scanners. Switching to CDK isn't a small retraining ask. This is the biggest argument against, and it's a real one. Worth direct discussion with Ketan and Cloud Engineering rather than assumed.

What's good about the hybrid

  • Locality preserved. App developers see infra alongside app code.
  • Conventions preserved. Cloud Engineering owns the constructs; consistency is enforced at the package level, not at the merge-review level.
  • Coordination tax shrinks. Day-to-day changes don't go through Cloud Engineering. Cross-cutting changes (a new tagging rule, a fix to the standard ECS wiring) are made once in the package; app repos pick them up at upgrade time.
  • Compliance story stays strong. Cloud Engineering can require apps to be on a minimum package version for audit purposes. The package itself becomes the audit point.

What's bad about the hybrid

  • Versioning becomes a real concern. When @sa/infra ships a breaking change, every app needs to upgrade. Same coordination problem the centralized repo currently absorbs, spread across N PRs. Mitigated by long deprecation windows and codemods, not eliminated.
  • Cloud Engineering's tooling investment changes substantially. Maintaining a TypeScript package in addition to (or instead of) Terraform modules is a real change for the team.
  • Drift between apps becomes possible. App A on v3.2, App B on v3.4 — different from the centralized repo's "everything is on the same commit at all times."
  • Cross-stack references are harder. Today a Terraform wrapper can reference module outputs directly; CDK across repos has to either pass IDs as strings (via SSM Parameter Store / CloudFormation outputs) or use a shared deployment construct.
Sandbox vs. Pre-Prod & Prod

Sandbox is in scope; prod is a separate conversation

The proposal commits the framework to sandbox infrastructure baked into the template. Pre-prod and prod are a different conversation with Cloud Engineering — different security posture, different network topology, different release controls.

What we'd build for sandbox

  • CDK infra/ in every project repo. Same template, same shape, same constructs.
  • Shared sandbox database. Hosted Postgres; each project creates its own schema rather than provisioning fresh RDS.
  • CI/CD via .github/workflows. First push deploys to a sandbox URL automatically.
  • The minimum substrate to make it work. A sandbox AWS account (or sub-account), a shared VPC, baseline IAM roles for the deploy pipeline, log/metric collection.

What's an open conversation

  • Does the same infra/ code carry into pre-prod and prod? Probably yes for the shape; almost certainly not for the values (different VPC, different secrets, different scaling). Whether that's a multi-environment CDK app or separate stacks is a design choice we'd make with Cloud Engineering.
  • Who owns the prod deploy decision? Sandbox is automatic. Prod isn't. The framework needs a clean gate — manual approval, change-management ticket, scheduled-release window — that fits Cloud Engineering's existing process.
  • How does @sa/infra (if we adopt it) get versioned for prod? Major-version upgrades may need a different cadence in prod than in sandbox.
  • Security review and audit hooks. Whatever scanners and policies Cloud Engineering runs against the existing infrastructure repo today need to apply to the new shape too. The format of "where Cloud Engineering presses 'go'" stays consistent.
Headline Take

What we'd recommend bringing to Cloud Engineering

The choice between options needs Cloud Engineering's voice, not just ours. Here's the framing we'd bring to that conversation.

For sandbox-tier work in the framework's first iteration: Option C (hybrid CDK) is the natural fit. It compounds with the template's TypeScript-first stack, gives AI Engineering ownership of routine infra changes, and keeps Cloud Engineering's audit hooks in place via the construct package.

For pre-prod and prod: Option A (Cloud Engineering's existing Terraform) is the realistic starting point — it asks nothing of Cloud Engineering's tooling. Option C as a longer-term direction (6–12 months) if Cloud Engineering wants to pull production into the same model.

Option B isn't on the table at any phase — Cloud Engineering's role depends on the central audit and convention point that pure per-app gives up.

What we need from Cloud Engineering to lock this in: their reaction to receiving AI-Eng-originated PRs against infrastructure code (whether centrally or per-app), any prior CDK consideration, comfort level with the versioning model of @sa/infra, and the existing PR/CI flow for the central infrastructure repo. We'd want this conversation early in the engagement.
