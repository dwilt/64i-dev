Appendix D
Where We Are Today

Today's deployment shape

SA's in-scope apps have a mix of deployment shapes today. The most common pattern — and the one that's most worth replacing — is a single EC2 instance with the application and a bundled Supabase stack running on it.

That shape isn't anyone's preference. It exists because Lovable generates a codebase tightly coupled to Supabase, and the engineering team isn't given enough time during the handoff to rework things — so dumping Supabase onto an EC2 instance in SA's AWS becomes the fastest path to a deployed app. That's a process problem expressing itself as an infrastructure problem. Phase 1 of this proposal directly addresses the time-budget side (the parallel engineering head start, the project briefing, the requirements doc); the deploy architecture below is what becomes possible once engineering isn't forced into the corner-cutting path.

The current shape carries three concrete costs that the framework's recommended architecture is designed to remove.

Increased cost
Each instance runs the application plus Postgres, the Supabase API gateway, auth, storage, edge-function runtime, and the rest of the bundled stack. That's a lot of overhead per app — multiplied across every app SA runs.
No horizontal scaling
Because the database lives on the instance, the app can only run one instance. Adding a second instance means two independent databases. There's no path to scale out for traffic, no path to zero-downtime deploys, no path to multi-AZ redundancy without restructuring the deploy entirely.
Maintenance burden
Every instance is a unique mini-stack to keep patched, upgraded, and monitored. The blast radius of an instance going down is total — app, DB, auth, and storage all gone together. Every app reinvents the same operational concerns.

These aren't criticisms of how SA got here — bundled-Supabase-on-EC2 is a reasonable answer to "get this thing running fast." But it doesn't scale to "we want to run dozens of apps reliably, in a way the team can support." The recommendation below is what SA's deploy shape looks like when the operational model is the priority instead.

The Recommendation

The shape we'd build toward

A small set of AWS primitives, used the same way in every project. The result: any SA engineer who has worked on one project knows how to operate any other.

The runtime topology

Two variants, decided in Phase 1. The topology below is the shape if the web app is a static SPA with a separate backend API — the pattern Lovable's older React + Vite output produces, and the one the prototype is on today. If Phase 1 selects TanStack Start (now Lovable's default on current plans), the UI and a server layer ship as one unit: there's no separate static-SPA target on S3, and the "Node service" below becomes the web app itself (UI + server functions in one ECS service) rather than a standalone API. The primitives don't change — Aurora, ECS, ALB, S3, CloudFront, Lambda — but the number and shape of the deploy targets do. We'd lock this with Cloud Engineering once the Phase 1 web-stack decision lands.

The rest of this section walks through each piece and the rationale.

Per-Primitive

What each piece does and why

Six pieces, each one solving a specific problem the bundled-Supabase-on-EC2 shape doesn't solve well. Stack picks below are starting points — Cloud Engineering's input refines them.

Database — shared Aurora (or RDS), project-scoped schemas

Every project that needs a database gets its own schema inside a shared Aurora cluster, rather than its own database instance. The application connects with credentials that grant access to its schema only — isolation enforced at the role level.

What this buys:

  • One thing to back up, patch, monitor, and tune. Per-project RDS instances multiply operational work by the number of projects; one shared cluster doesn't.
  • Drops the per-project provisioning tax to near zero. Creating a new project means running a migration to create a schema, not provisioning AWS resources.
  • Encryption, multi-AZ, snapshots, and audit are configured once. Every project inherits them automatically.
  • Real horizontal scaling for the apps. Once the database is no longer co-located with the app, multiple ECS tasks can serve traffic against the same data — the single-instance ceiling disappears.

Aurora vs. RDS — an evaluation for Cloud Engineering. Aurora's storage model (separating compute from storage, fast scale-out reads) is the right answer if SA expects significant read traffic or wants serverless-v2-style autoscaling on the DB itself. Plain RDS Postgres is cheaper and simpler if the workload doesn't warrant Aurora's premium. We'd land on one with Ketan's team.

What the framework expects from the schema model. The template's CDK includes a small migration step that creates the project's schema and a least-privilege application role on first deploy. The app's connection string targets that schema. No project ever directly creates an RDS instance; all DB provisioning runs through the shared-cluster substrate Cloud Engineering owns.

APIs — ECS Fargate with autoscaling

The Node service (framework selected in Phase 1 — Fastify, NestJS, or TanStack Start server functions) and the Python API (FastAPI, when present) deploy to ECS Fargate. One task definition per service. Application Load Balancer in front, routing by path. Target-tracking autoscaling on CPU and request count.

Note on the service count. In the static-SPA variant, the Node API is a service distinct from the web app — two-plus services (web build on S3, Node API on ECS, optional Python API). If Phase 1 selects TanStack Start, the web app is the Node service: UI and server functions run together in one ECS task, so there's no separate static-SPA target and the "Node API" and "frontend" collapse into a single web/server service (plus the optional Python API).

What this buys:

  • No instance management. Fargate runs the container; SA doesn't run the EC2 host underneath. One less operational concern per project.
  • Real horizontal scaling. The same task definition can run 1 task or 50, scaled by traffic. Zero-downtime deploys via ECS's rolling update.
  • Same operational shape across projects. Every project's APIs look identical in CloudWatch, in IAM, in the deploy pipeline. Engineering knows how to operate any project from day one.
  • Python and Node coexist cleanly. They're different task definitions in the same cluster; the ALB routes between them. Projects that don't need the Python API just don't deploy that task.

What ships in the template's CDK: the ECS service, task definition, ALB listener rules, target groups, log group, IAM execution role, and the autoscaling policy — all parameterized by project. The numbers (CPU/memory/min-max instance count) start at sensible defaults and are tuned per project as real traffic shows up.

Frontend — S3 behind CloudFront (or SA's CDN)

This applies in the static-SPA variant. If the web app is a static SPA, its build output goes to a per-project S3 bucket, with CloudFront (or whatever CDN SA already uses) in front for TLS, caching, and the project's custom domain. Under TanStack Start SSR, the web app runs as a server (the ECS Node service above), not a static bundle — there's no S3/CloudFront frontend target; CloudFront, if used, sits in front of the ALB for caching.

What this buys:

  • Cheapest possible static hosting. S3 storage costs are trivial; CloudFront handles the request load.
  • No application server for the SPA. The browser fetches an immutable bundle; backend traffic goes straight to the ALB.
  • Cache-bust on deploy. The deploy pipeline invalidates the CloudFront distribution after pushing new assets to S3. The user gets the new version on the next page load.

If Cloud Engineering already has a frontend hosting story they prefer (Amplify, a different CDN, a shared distribution model), we'd align to that during the engagement. The shape above is the default; it isn't a hard requirement.

File storage — per-project S3 buckets

When a project needs file storage — uploads, generated artifacts, document attachments — it gets its own dedicated S3 bucket. Naming convention along the lines of saai-<project>-files-<env>. Encryption at rest, versioning enabled, lifecycle rules for cost control where appropriate.

Why per-project rather than shared: bucket-level access control is the cleanest isolation primitive AWS offers. Per-project buckets mean each project's IAM role only needs access to its own bucket; cross-project blast radius is zero. The cost difference between one shared bucket and N per-project buckets is essentially nothing.

Access from the Node API and Python API is via the AWS SDK using the task's IAM role — no static credentials, no per-project secrets management for storage.

Scheduled and async work — Lambda + EventBridge

Today the monorepo template doesn't have a slot for Lambda functions. Adding one is straightforward; the question worth being honest about is how Lambdas are used in a project, and that decision is one we'd make per-use-case rather than committing to a single pattern in advance.

Two patterns we'd evaluate when the first project needs them:

Pattern A — Lambda invoked from the API via the AWS SDK

The Node or Python API invokes a Lambda directly (synchronously or asynchronously) for jobs that are heavier or longer-running than a request handler should hold open. The Lambda has its own IAM role, runs in isolation, and reports back via SDK response or by writing to a queue/DB. The API stays in front; the Lambda is implementation detail.

When this is the right pattern: "I want my API to kick off a job and not wait" — file processing, report generation, embedding generation, anything bounded but not request-shaped. Lighter to set up; the Lambda doesn't need to be addressable from outside.

Pattern B — Lambda behind API Gateway

The Lambda is exposed as an HTTP endpoint via API Gateway. Useful when something genuinely needs to be HTTP-callable from outside the SA stack — webhooks from third parties, scheduled jobs that are easier to debug as a request, callable endpoints that shouldn't go through the always-on ECS API.

When this is the right pattern: when "this needs a URL of its own" is part of the requirement. Heavier to set up than Pattern A; warranted when the requirement actually demands it.

Plus the obvious third — scheduled Lambdas via EventBridge rules — for cron-style work (a nightly report, a periodic data refresh, a cleanup job). This is the lowest-ceremony slot and the one most projects will reach for first.

What we'd add to the monorepo template during the engagement: a slot at apps/lambdas/ (or similar) for the Lambda code, a CDK construct in infra/ for deploying it, and a RULES.md entry covering when to use Pattern A vs. Pattern B vs. EventBridge-only. The patterns become reusable from there; future projects pick the slot up for free.

Status: roadmap. The Lambda pattern isn't in today's prototype. We'd build the first one against a real project's needs during the engagement, and let the actual use case shape the construct rather than over-designing in advance.

Observability — CloudWatch (or whatever Cloud Engineering uses today)

Logs, metrics, and traces from every primitive above land in CloudWatch by default. If SA has a different observability target (Datadog, New Relic, a custom Grafana stack), we'd point things there instead — the framework's job is to wire up the integration consistently, not to dictate the destination.

The template's CDK creates the log groups, sets retention policies, and emits the standard application metrics (request count, error rate, latency percentiles, task CPU/memory utilization). Per-project dashboards are a small follow-on; the data is there from day one.

Sandbox vs. Prod

What carries forward and what doesn't

The shape above is what the framework commits to for sandbox. Pre-prod and prod use the same shape with different configuration — but the gap between "sandbox is designed" and "prod is decided" is real and worth being explicit about.

What carries forward unchanged

  • The primitives. Aurora, ECS, ALB, S3, CloudFront, Lambda, EventBridge. Same set of building blocks in every environment.
  • The project's shape. Whatever shape an app has in sandbox — a static SPA plus a separate Node API, or a unified TanStack Start web/server, plus an optional Python API — it has the same shape in prod. Engineers operate it the same way.
  • The IaC structure. Same CDK app, same constructs, same deploy pipeline — just pointed at different environments.

What changes between environments

  • VPC and account model. Sandbox can share a VPC across many projects; prod likely has stricter isolation, possibly per-account separation for blast-radius reasons. To work out with Cloud Engineering.
  • Scaling minimums. Sandbox can run a single task at idle; prod usually wants at least two for redundancy.
  • Deploy gate. Sandbox deploys on every push. Prod has a manual approval, a release window, a change-management ticket — whatever Cloud Engineering's existing release model requires.
  • The shared Aurora cluster itself. Whether prod uses one shared cluster the same way sandbox does, or each "tier" of app gets its own cluster, or some apps justify their own dedicated cluster — that's a Cloud Engineering decision.
  • Secrets and config. Sandbox can use sensible defaults; prod uses Secrets Manager / Parameter Store with proper rotation. The template can wire this up consistently as long as we know what's expected.
Open Questions

For Cloud Engineering

The shape above is a defensible default. The decisions below specifically need Cloud Engineering's voice — they all touch operational concerns we're not in a position to commit to without them.

  • Aurora vs. RDS Postgres for the shared cluster, and whether that cluster is a new dedicated thing for AI Engineering apps or an extension of something Cloud Engineering already runs.
  • ECS cluster shape — one cluster per environment with all projects' services running inside it, or one cluster per project? Both are defensible; the trade-off is operational simplicity vs. blast-radius isolation.
  • CDN choice and the custom-domain story. CloudFront is the default assumption; if SA's existing distribution model is different, the template should match it.
  • Observability target. CloudWatch by default; whatever Cloud Engineering already uses, ideally.
  • Account topology. One sandbox account, one prod account; per-environment accounts; per-team accounts. The template's CDK assumes one but isn't locked to it.
  • The prod deploy gate. What flavor of approval / change management does prod require — and where should the framework's pipeline plug into it.
